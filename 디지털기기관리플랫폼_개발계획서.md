## 1. 개발 개요

본 플랫폼은 초등학교 내 디지털 기기를 체계적으로 관리하기 위해 디지털정보부장 주도로 기획된 웹 기반 관리 시스템입니다.  
학급별 기기 현황 파악, 고장 신고 및 처리, 공유 기기 대여·반납을 통합 관리하여 업무 효율을 높이고 기기 운용 투명성을 확보합니다.

### 1-1. 개발 목표

- 교사가 간편하게 로그인하여 학급 기기 현황을 등록·관리
- 고장 기기 신고 및 처리 현황을 실시간으로 관리자에게 전달
- 학교 공유 디지털 기기의 대여 및 반납 현황을 투명하게 공유
- 디지털정보부장(관리자)이 전체 현황을 한눈에 파악

### 1-2. 기술 스택

| 구분 | 기술 | 역할 |
|------|------|------|
| 프론트엔드 | React + Next.js 16 (App Router) | UI 렌더링, 페이지 라우팅, SSR/CSR 처리 |
| 백엔드/DB | Supabase (PostgreSQL) | 인증, 데이터 저장 |
| 인증 | Supabase Auth + localStorage | 관리자 이메일 로그인 / 교사 학교코드+학급 선택 로그인 |
| 스타일링 | Tailwind CSS v4 | 반응형 UI, 모바일 지원 |
| 배포 | Vercel (GitHub 연동) | Next.js 최적화 배포 환경 |

## 2. 사용자 구분 및 로그인 설계

### 2-1. 사용자 역할

| 역할 | 설명 | 권한 |
|------|------|------|
| 관리자 (디지털정보부장) | 플랫폼 최고 관리자 | 전체 현황 조회, 고장 처리, 기기 등록, 학급 등록·기기 수량 관리, 대여 반납 최종 승인 |
| 교사 (학급 담당) | 각 학급 담임 교사 | 본인 학급 기기 등록, 고장 신고, 대여 신청·반납 요청 |

### 2-2. 로그인 방식

#### 관리자 로그인 / 회원가입
- `/login/admin` 페이지에서 **로그인 / 회원가입** 탭으로 구분
- 회원가입: 이메일 + 비밀번호 입력 → Supabase Auth 계정 생성 → 학교명 + 학교코드 초기 설정 후 대시보드 이동
- 로그인: 이메일/비밀번호 입력 → 관리자 대시보드로 이동
- JWT 토큰 기반 세션 유지
- 로그인 후 `/admin/settings`에서 학교명·학교코드 언제든 변경 가능

#### 교사 로그인 (2단계)
- **1단계**: 학교코드 입력 → Supabase `school_config` 조회로 유효성 검증
- **2단계**: 관리자가 사전 등록한 학급 목록이 카드 형식으로 표시 → 본인 학급 선택
- 별도 회원가입 없이 세션 생성 (localStorage 기반 `TeacherSession`)
- 학급이 하나도 없으면 "관리자에게 학급 등록을 요청하세요" 안내 표시

### 2-3. Supabase 인증 구조

교사 로그인은 localStorage에 `TeacherSession`(classroomId, className, teacherName)을 저장하는 커스텀 방식을 사용합니다.  
관리자는 Supabase Auth를 통해 이메일 인증 기반으로 로그인합니다.

| 테이블 | 주요 컬럼 | 설명 |
|--------|-----------|------|
| `school_config` | school_code, school_name | 학교코드 및 학교명 저장 |
| `classrooms` | id, class_name, teacher_name, created_at | 학급 정보 (관리자가 사전 등록) |
| `users` (Supabase Auth) | id, email | 관리자 계정 관리 |

## 3. 기능 1 — 학급별 기기 관리

### 3-1. 기능 개요

각 학급이 보유한 디지털 기기(크롬북, 펜, 마우스 등)의 수량을 직접 등록하고 관리합니다.  
기기에 고장이 발생하면 교사가 신고하고, 관리자가 접수 및 처리 상태를 업데이트합니다.

### 3-2. 관리자 화면 — 학급 관리

- `/admin/classrooms`에서 등록된 학급을 **그리드 카드** 형식으로 표시
- 카드에는 학급명, 담당 교사명, 기기 종류별 수량 요약, 총 기기 수 표시
- 카드 클릭 시 상세 모달에서 **기기 수량 직접 편집** 가능 (관리자 권한)
- 학급 추가: 학급명 + 담당 교사명 입력
- 학급 삭제: 카드 hover 시 X 버튼 표시 → 확인 후 삭제 (관련 데이터 cascade 삭제)
- 기기 종류 목록은 `기기 종류 설정` 메뉴에서 관리자가 사전 설정

### 3-3. 교사 화면 — 기기 등록

- 학급 로그인 후 **내 학급 기기 현황** 페이지 접근
- 기기 종류(크롬북, 펜, 마우스 등) 및 보유 수량 입력/수정
- 기기 목록은 테이블 형식으로 표시, **수량 수정** 버튼으로 편집 모드 전환
- 관리자가 기기 종류 목록을 사전 설정하여 선택 방식으로 제공

### 3-4. 교사 화면 — 고장 신고

- 보유 기기 목록에서 **고장 신고** 버튼 클릭
- 고장 기기 종류, 고장 수량, 고장 내용(선택) 입력
- 신고 후 '신고 완료' 상태로 표시되며 재신고 방지
- **내 신고 내역** 탭에서 신고 이력 및 처리 상태 확인 가능

### 3-5. 관리자 화면 — 고장 신고 관리

- **고장 신고 관리** 탭에서 전체 학급의 신고 목록 조회
- 신고 목록 표시 항목: 신고 날짜 / 학급명 / 기기 종류 / 고장 수량 / 처리 상태
- 처리 상태 3단계: `접수 대기` → `수리 중` → `처리 완료`
- 각 신고 건에 **처리** 버튼 클릭 시 상태 업데이트
- 필터 기능: 처리 상태별 필터링

### 3-6. 데이터베이스 설계

| 테이블 | 주요 컬럼 | 설명 |
|--------|-----------|------|
| `devices` | id, device_type, description | 기기 종류 마스터 (관리자 설정) |
| `classroom_devices` | classroom_id, device_id, quantity | 학급별 기기 보유 수량 |
| `repair_reports` | id, classroom_id, device_id, quantity, status, reported_at, resolved_at | 고장 신고 및 처리 이력 |

### 3-7. 화면 흐름도

| 단계 | 교사 행동 | 시스템 동작 | 관리자 반응 |
|------|-----------|-------------|-------------|
| 1 | 학급 로그인 | 학급 세션 생성 | — |
| 2 | 기기 수량 등록/수정 | classroom_devices 저장 | 학급 카드 기기 수 즉시 반영 |
| 3 | 고장 신고 입력 | repair_reports 생성 (접수 대기) | 신고 목록에 즉시 표시 |
| 4 | 신고 내역 확인 | 처리 상태 표시 | — |
| 5 | — | — | 처리 버튼 클릭 → 상태 변경 |
| 6 | 처리 완료 알림 확인 | 완료 상태 표시 | — |

## 4. 기능 2 — 학교 공유 기기 대여·반납

### 4-1. 기능 개요

관리자가 학교 공용 디지털 기기(태블릿, 빔 프로젝터, 노트북 등)를 플랫폼에 등록하면,  
교사들이 원하는 수량을 직접 대여 신청하고 반납 요청할 수 있습니다.  
반납은 교사의 요청 후 관리자가 최종 확인·처리하는 **2단계 반납 절차**를 따릅니다.

### 4-2. 관리자 화면 — 공유 기기 등록

- **공유 기기 관리** 메뉴에서 기기 종류 및 총 수량 등록
- 기기별 현재 대여 가능 수량 자동 계산 및 표시
- 기기 추가·수정·비활성화·**삭제** 기능 제공

### 4-3. 교사 화면 — 대여 신청

- **기기 대여** 메뉴에서 현재 대여 가능한 기기 목록 확인
- 기기 종류 선택 후 대여 수량 입력
- 대여 신청 시 자동으로 대여 날짜 및 학급명 기록
- 대여 수량이 재고 초과 시 신청 불가 처리

### 4-4. 전체 대여 현황 (공개)

- 로그인한 모든 교사가 전체 대여 목록 조회 가능
- 표시 항목: **대여 날짜 / 학급명 / 기기 종류 / 대여 수량 / 상태**
- `대여 중` / `반납 요청 중` 항목을 함께 표시

### 4-5. 반납 절차 (2단계)

| 단계 | 행위자 | 행동 | 상태 변화 | 재고 변화 |
|------|--------|------|-----------|-----------|
| 1 | 교사 | **반납 요청** 버튼 클릭 | `대여 중` → `반납 요청 중` | 변화 없음 |
| 2 | 관리자 | **반납 처리** 버튼 클릭 | `반납 요청 중` → `반납 완료` | available_quantity 복구 |

- 교사 화면: `반납 요청 중` 항목은 "반납 요청됨" 텍스트로 표시 (재요청 불가)
- 관리자 화면: 기본 탭이 **반납 요청** 탭으로 열려 대기 건 즉시 확인 가능
- 반납 요청 행은 연보라 배경으로 강조 표시

### 4-6. 데이터베이스 설계

| 테이블 | 주요 컬럼 | 설명 |
|--------|-----------|------|
| `shared_devices` | id, device_name, total_quantity, available_quantity, is_active | 학교 공유 기기 목록 및 재고 |
| `rentals` | id, classroom_id, device_id, quantity, rented_at, returned_at, status | 대여·반납 이력 관리 |

- `rentals.status` 허용값: `'대여 중'` / `'반납 요청 중'` / `'반납 완료'`

### 4-7. 화면 흐름도

| 단계 | 행위자 | 행동 | 시스템 동작 |
|------|--------|------|-------------|
| 1 | 관리자 | 공유 기기 등록 (종류/수량) | shared_devices 생성 |
| 2 | 교사 | 대여 목록 조회 | available_quantity 실시간 표시 |
| 3 | 교사 | 기기 종류 + 수량 선택 후 대여 신청 | rentals 생성, 재고 차감 |
| 4 | 전체 교사 | 대여 현황 목록 확인 | 전체 대여·반납 요청 현황 표시 |
| 5 | 교사 | 내 대여 목록에서 반납 요청 버튼 클릭 | status → '반납 요청 중' |
| 6 | 관리자 | 반납 요청 탭에서 확인 후 반납 처리 | status → '반납 완료', 재고 복구 |

## 5. 시스템 아키텍처 및 화면 구성

### 5-1. 전체 페이지 구조

| 경로 (URL) | 페이지명 | 접근 권한 |
|------------|----------|-----------|
| `/` | 메인 (로그인 선택) | 모두 |
| `/login/admin` | 관리자 로그인 / 회원가입 | 모두 |
| `/login/teacher` | 교사 로그인 (학교코드 → 학급 선택) | 모두 |
| `/admin/dashboard` | 관리자 대시보드 | 관리자 |
| `/admin/settings` | 학교 설정 (학교코드 관리) | 관리자 |
| `/admin/classrooms` | 학급 등록·관리 (그리드 카드, 기기 수량 편집) | 관리자 |
| `/admin/repairs` | 고장 신고 관리 | 관리자 |
| `/admin/device-types` | 기기 종류 설정 | 관리자 |
| `/admin/devices` | 공유 기기 관리 (추가·수정·삭제·비활성화) | 관리자 |
| `/admin/rentals` | 전체 대여 현황 (반납 요청 탭 기본) | 관리자 |
| `/teacher/dashboard` | 교사 대시보드 | 교사 |
| `/teacher/my-devices` | 내 학급 기기 현황 | 교사 |
| `/teacher/repair` | 고장 신고 | 교사 |
| `/teacher/rentals` | 기기 대여 신청 / 반납 요청 | 교사 |

### 5-2. Supabase Row Level Security (RLS) 설계

- 현재 모든 테이블은 `for all using (true) with check (true)` 정책으로 전체 허용
- `school_config`: 읽기/쓰기 모두 허용 (관리자 회원가입 시 anon 상태로 insert 필요)
- 추후 권한 세분화 시 `classroom_id` 기반 RLS 적용 예정

### 5-3. 전체 DB 스키마 요약

```sql
-- 학교 설정
school_config       (id, school_name, school_code)

-- 학급
classrooms          (id, class_name, teacher_name, created_at)

-- 기기 종류 마스터
devices             (id, device_type, description)

-- 학급별 기기 보유 현황
classroom_devices   (id, classroom_id, device_id, quantity)
  unique(classroom_id, device_id)

-- 고장 신고
repair_reports      (id, classroom_id, device_id, quantity,
                     description, status, reported_at, resolved_at)
  status: '접수 대기' | '수리 중' | '처리 완료'

-- 공유 기기
shared_devices      (id, device_name, total_quantity, available_quantity, is_active)

-- 대여·반납
rentals             (id, classroom_id, device_id, quantity,
                     rented_at, returned_at, status)
  status: '대여 중' | '반납 요청 중' | '반납 완료'
```

## 7. 기대 효과

- **기기 현황 파악 시간 단축** — 학급별 기기 현황을 그리드 카드로 한눈에 파악, 분기별 재고 조사 업무 간소화
- **고장 처리 투명성 향상** — 신고~처리 전 과정이 기록되어 누락 없는 기기 유지보수 가능
- **대여 관리 효율화** — 수기 대여 장부 대체, 2단계 반납 절차로 분실·미반납 방지
- **데이터 기반 의사결정** — 기기 고장 빈도, 대여 패턴 데이터를 기반으로 기기 구매 계획 수립
- **교사 업무 부담 감소** — 학교코드+학급 선택 방식으로 접근성 향상, 별도 교육 없이 사용 가능
